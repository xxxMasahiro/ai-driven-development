# AI駆動開発14日版レッスンガイド

このファイルは、14日版レッスンの学習者向けガイドです。
既存の7日版ガイドは変更せず、拡張版として使います。

## 学ぶこと

```text
AIエージェントに頼む順番
エージェントとの壁打ちで目的を整理し、必要な情報を効率よく引き出す方法
要件、仕様、実装計画を分ける理由
TASK_TRACKER.mdとHANDOFF.mdで作業を引き継ぐ方法
GitとGitHubでローカルとリモートを同期する方法
CIで変更を検査する方法
Playwrightで画面操作を自動確認する方法
複数サブエージェントや役割プロンプトで多角的に見る方法
スキルとMCPを小さく使う方法
```

## 学習モード

```text
A: 詳細解説が必要
Git、GitHub、CI、PR、Playwright、サブエージェント、スキル、MCPなどを、目的と便利さから説明する。

B: 解説は補足程度
作業の直前に、専門用語や技術用語だけを短く補足する。

C: 解説不要でワークフローのみ
復習向け。説明は省き、作業、確認、同期ゲートだけを進める。
```

初回はAを推奨します。
2周目以降はBまたはCにすると、同じ教材を復習用ワークフローとして使えます。
学習モードはレッスン中いつでも `./tools/lesson14 学習モード <A|B|C>` で切り替えられます。

## 進行ルール

```text
各レッスン項目は一問一答で進める。
次の項目へ進む前に、必ずユーザーの承認を得る。
エージェントは、現在地、目的、これから行うこと、必要なコピペ用プロンプトを提示してから止まる。
```

## Step 12/14からStep 13/14の学び方

サブエージェント、スキル、MCPは、単にエージェントが裏側で処理して終わらせません。

```text
1. 何のために使うのかを説明する
2. 何が便利になるのかを説明する
3. どのCLI画面で何を入力するのかを提示する
4. 学習者がコピペして実行できるプロンプトを出す
5. 実行結果を見て、次に進むか確認する
```

MCPを本格実装しない場合は、「MCP風の最小リソース」と「本物のMCPサーバー」の違いを明示します。
MCPに入る前には、連携先、入力、出力、最小範囲、便利になる点、変更されるファイル、確認方法を学習者に示します。
実装や代替スクリプト作成は、学習者がそのまま使える依頼文を確認し、開始を承認した後にだけ進めます。

## 対話と壁打ち

このレッスンの中核は、開発者が目的を与え、エージェントとの対話でその目的を具体的な開発行動へ変換することです。
学習者は、何を作りたいのか、なぜ必要なのか、どの制約があるのか、何を優先するのかをエージェントへ伝えます。
エージェントは、要件、仕様、実装計画、検証方法、リスク、次の一手を質問と提案で引き出します。

実践では、教材CLIと成果物CLIを分けたまま進めます。
教材CLIでは現在地、説明、承認、同期ゲートを扱います。
成果物CLIでは、実際のプロンプト入力、実装、テスト、Git操作を体験します。

サブエージェントは、目的達成のために複数の観点が必要な場面で使います。
例: レビュー担当、テスト担当、UX担当、セキュリティ担当、同期確認担当。
統括役は、各観点の指摘をそのまま混ぜず、目的、優先度、実装コスト、リスクに照らして採用、保留、却下を判断します。

## 完了後の改善サイクル

14日版完了後は、同じ開発フローで次の改善サイクルへ進めます。
任意の成果物を設定して開発したい場合は、自由開発モードを使います。

```text
1. 見た目改善: レイアウト、スマホ表示、完了タスクの見え方
2. 機能追加: 保存、削除、編集、検索、絞り込み
3. 連携追加: Googleカレンダーなど外部サービスとの連携案を設計する
4. PRで小さく実装し、CIとE2Eで確認する
5. TASK_TRACKER.mdとHANDOFF.mdを更新する
```

```bash
./tools/free-development start
./tools/free-development gate
```

発展的にチーム開発とDockerを学ぶ場合:

```bash
./tools/team-development start
./tools/team-development gate
```

## CLI画面の分離

成果物リポジトリの開発に入る前には、別画面でUbuntu/WSL CLIを起動します。
学習用リポジトリ `$HOME/projects/ai-driven-development-lesson/` と、成果物リポジトリ `$HOME/projects/task-tracker-repository/` を別々のCLI画面で扱うことで、Git管理の混線を防ぎます。

## 学習者が使う主なコマンド

```bash
./tools/lesson14 status
./tools/lesson14 初期化 --confirm
./tools/lesson14 開始位置 <step_id> --confirm
./tools/lesson14 学習モード <A|B|C>
./tools/lesson14 承認 start <step_id> "ユーザーがこの項目の開始を承認した"
./tools/lesson14 開始 <step_id>
./tools/lesson14 承認 pass <step_id> "ユーザーがこの項目の通過を承認した"
./tools/lesson14 通過 <step_id> "メモ"
./tools/roadmap status
./tools/helpdesk 相談 "分からないこと"
./tools/helpdesk 解決 "分かったこと"
```

## 困ったとき

```text
分からない場合は、分からないと答える
エラーが出た場合は、出力をそのままエージェントに見せる
未来項目を進めたい場合は、現在地に戻る
急いでいる場合でも、同期ゲートは省略しない
```
